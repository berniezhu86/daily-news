## 7. 排序总原则

排序不是单纯按权威，也不是单纯按热度。正确优先级如下：

```text
发布时间 / 时效层级 > 新闻重要性 > 来源权威 > 热度 > 摘要质量 > 曝光轮换
```

### 7.1 时效层级是硬门槛

新闻先按时间分层：

| 层级 | 时间 | 排序意义 |
|---|---|---|
| S | 24 小时内 | 首页和前几页主力 |
| A | 24-48 小时 | 只做保底和补充 |
| B | 48-72 小时 | 原则上不进首页，除非是持续重大事件的权威更新 |
| C | 超过 72 小时 | 默认淘汰或进入池底 |

严禁让 48 小时前的旧权威稿长期压住当天新闻。

### 7.2 高层时政要闻规则

24 小时内的高层时政要闻权重最高。

命中范围：

- 习近平
- 国家主席
- 中共中央总书记
- 中央军委主席
- 李强且明确为国务院总理/主持召开/会见/出席
- 国务院常务会议
- 中共中央政治局
- 全国人大常委会
- 全国政协
- 外交部、国台办等政府高层正式发布

排序规则：

1. 24 小时内 + 权威来源 + 国家元首/政府高层：首屏优先。
2. 24-48 小时：只做保底，不得挤掉大量当天热点。
3. 超过 48 小时：降权，不进首页前几页。
4. 如果 24 小时内高层时政不足，就把当天其他权威热点、重大突发新进展补上。

### 7.3 重大突发事件规则

重大突发包括：

- 地震、强震、余震、震级
- 洪水、暴雨、台风、山洪、泥石流、滑坡、内涝
- 火灾、爆炸、坍塌、沉船、矿难
- 且伴随死亡人数、遇难、受伤、失踪、救援、获救、应急响应、红/橙/黄预警等信号

重大突发只做“补位”，不能变成最高权重。

硬性限制：

1. 首页前 20 条重大突发最多 2 条。
2. 同一事件只保留 1 条最重要进展。
3. 24 小时内重大进展优先于 24-48 小时旧进展。
4. “死亡人数升至”“已有多少人获救”“救援最新情况”优先于“哀悼七天”“各方表态”。
5. 普通军事袭击、国际冲突、战争新闻不要误判成灾害，除非它本身是灾难救援类事件。

例子：

- `委内瑞拉地震死亡人数升至2954人，已有6462人获救`：可作为重大突发补位。
- `委内瑞拉为地震遇难者哀悼七天`：属于旧延伸稿，不能靠前。
- `广西发布洪水黄色预警`：可补位，但不能挤掉大量当天时政/国际新闻。

### 7.4 权威来源加权规则

权威来源加分，但不能越过时效层级。

高权威：

- 新华社、新华网
- 人民日报、人民网
- 央视新闻、央视网、央广网
- 中国政府网
- 外交部、国防部、商务部、财政部、发改委等部委
- 中国新闻网、中新社
- 财经类：财联社、证券时报、中国证券报、上海证券报、第一财经、21世纪经济报道

低质量/降权来源：

- 自媒体、公众号、博客、个人号
- 标题党、营销号、荐股号
- 来源名明显不严肃或不可核验
- AI 科技中的社交媒体转述，除非是官方账号或行业一手消息

规则：

```text
同一时效层级内：权威来源优先。
跨时效层级：当天新闻优先，旧权威不能压住当天新闻。
```

### 7.5 去重与合并规则

同一事件多来源时，只保留 1 条，最多 2 条。

保留优先级：

1. 权威来源。
2. 发布时间更新。
3. 摘要更完整。
4. 标题更清楚。
5. 有原文链接。

重复判断方式：

- URL 去参数后相同。
- 标题高度相似。
- 标题不同但事件主体相同，例如“委内瑞拉地震死亡人数升至...”和“外媒：委内瑞拉地震死亡人数...”属于同事件。

### 7.6 曝光轮换规则

目的：避免用户每次打开都看到同一批新闻。

规则：

1. 普通新闻多次出现在首屏后降权。
2. 高层时政、重大突发可以减少曝光惩罚，但不能完全豁免。
3. 同一主题连续多次进入首屏时，保留最新/最权威一条，其余后移。
4. `news_refresh_state.json` 保存曝光状态，不能随意删除。

## 8. 分板块审核规则

### 8.1 热点页

热点页是合并流，用户第一眼看到的页面，质量要求最高。

目标结构：

1. 24 小时内高层时政/权威新闻。
2. 24 小时内国际重要新闻。
3. 当天重大突发新进展，最多 2 条补位。
4. 24 小时内财经、AI、民生、社会热点。
5. 24-48 小时新闻只做补充。

禁止：

- 前几页全是灾害。
- 前几页全是自媒体。
- 前几页出现大量 48 小时前旧闻。
- 同一事件连续刷屏。

### 8.2 财经市场新闻

保留：

- A 股、港股、美股、全球市场
- 央行、利率、汇率、债券
- 上市公司财报、业绩、回购、分红、IPO
- 产业政策与资本市场相关内容

过滤：

- 与财经无关的文旅、体育、娱乐。
- 纯 ETF 营销软文。
- “值得关注”“布局机会”等荐股型标题。
- 无数据、无事实的观点稿。

财经市场新闻不能为 0。若为 0，必须停止提交并回查 `news_pool.json` 与 `main_news_refresh.py` 的过滤条件。

### 8.3 足球新闻

保留：

- 河南队官方消息。
- 河南队赛程、战绩、伤停、训练、发布会。
- 中超积分、争冠、保级、裁判、足协公告。
- 可信体育媒体消息。

过滤：

- 纯球迷号。
- 与中超/河南无关的国际足球泛新闻。
- 啤酒节、演唱会、美食、文旅等蹭足球热度内容。

足球动态容量：河南队/中超动态不宜过多旧闻，超过 7 天默认降权。

### 8.4 娱乐新闻

保留：

- 电影、电视剧、综艺、音乐、演唱会。
- 票房、热度榜、定档、首映、主创、平台播出信息。

过滤：

- 体育、财经、AI、灾害、法治等错分类。
- 低质八卦、营销号、无实质信息标题。
- 争议性但无事实来源的爆料。

### 8.5 AI 科技新闻

保留：

- AI 模型、芯片、机器人、自动驾驶、科技公司产品发布。
- 官方博客、研究论文、权威科技媒体。

过滤：

- 纯 X/Twitter 转述且无上下文。
- 标题过长、摘要缺失、无法理解的技术碎片。
- 与 AI 科技无关的 ETF 营销稿。

## 9. `main_news_refresh.py` 接手说明

`main_news_refresh.py` 是每日审核排序的核心脚本。它主要做：

1. 读取 `news_pool.json`。
2. 按栏目过滤低质/错分类新闻。
3. 去重合并。
4. 计算质量分。
5. 根据高层时政、权威来源、发布时间、曝光状态排序。
6. 写回 `index.html` 对应数组。
7. 更新 `news_refresh_state.json` 和 `news_refresh_report.json`。

马维斯接手后必须重点检查以下函数：

| 函数 | 作用 | 风险 |
|---|---|---|
| `is_authoritative()` | 判断权威来源 | 来源名单过宽会把普通媒体抬太高 |
| `is_low_quality()` | 判断低质来源 | 名单过宽会误杀正常新闻 |
| `is_high_politics()` | 判断高层时政 | 不能把所有带“国务院/外交部”的普通新闻都置顶 |
| `is_priority_high_politics()` | 判断可优先时政 | 必须结合 24/48 小时时效规则 |
| `is_breaking_disaster()` | 判断重大突发 | 不能把战争袭击、普通事故都当灾害最高权重 |
| `item_quality()` | 算分 | 不允许灾害加分超过 24 小时时政 |
| `sort_section()` | 最终排序键 | 发布时间必须是第二重要因素，不能只按权重 |
| `dedupe_section()` | 去重 | 相似度阈值过低会误合并，不宜低于 0.9 |
| `section_item_allowed()` | 分栏目过滤 | 财经/娱乐/足球最容易被误杀或错放 |

### 9.1 必须修正/保持的排序方向

如果脚本中出现类似逻辑，要立即修正：

```python
# 危险：灾害直接 pin 到最高，容易导致首页全是灾害
disaster_pin = 1 if is_breaking_disaster(item) and is_authoritative(item) and age <= 48 else 0
return (top_leader_pin, politics_pin, disaster_pin, score, authority, -age)
```

正确方向：

```text
24小时内高层时政 > 24小时内权威热点 > 24小时内重大突发补位 > 24-48小时高层时政保底 > 其他新闻
```

重大突发只能限量补位，不能作为全局最高 pin。

### 9.2 每次跑完要看 `news_refresh_report.json`

至少检查：

- 各板块新闻数量是否为 0。
- 财经市场是否被过滤空。
- 娱乐是否混入体育/财经/灾害。
- 足球是否混入文旅/演唱会/泛体育。
- 重复合并数量是否异常过高。
- 首屏高层时政是否超过 48 小时。
- 首页前 20 条灾害类是否超过 2 条。

## 10. 前端排序函数接手说明

`index.html` 内还有一套前端抗疲劳排序函数，主要用于页面打开时动态排序：

- `parseNewsTimeValue(item)`：解析新闻时间。
- `getSourceAuthorityLevel(item)`：前端来源权威级别。
- `isHighPoliticsNews(item)`：前端高层时政判断。
- `isBreakingDisasterNews(item)`：前端重大突发判断。
- `getNewsAuthorityPriority(categoryKey, item, ageHours)`：权威/时政优先级。
- `getNewsRankTier(categoryKey, item, ageHours, authorityPriority)`：时效层级。
- `buildFreshNewsOrder(categoryKey, arr, extraArr)`：最终前端排序。
- `balanceBreakingDisasterItems(ordered, categoryKey)`：重大突发限量补位。

必须保持的前端验收指标：

1. 首页前 20 条中，24 小时内新闻应占主要比例；当天新闻充足时应接近全部。
2. 首页前 20 条中，重大突发不超过 2 条。
3. 24-48 小时时政只做保底，不能压住当天新闻。
4. 超过 48 小时的普通旧闻不能进入首页前几页。
5. 低质量自媒体不能靠前。

如果发现 `index.html` 中出现以下字样，要警惕：

```js
return 5.2
freshnessScore * 2.5
disaster_pin
```

这些曾导致灾害类新闻霸屏，除非你明确重新设计过限量补位，否则不要保留。

- 权威发布只显示国家元首/政府高层/重大突发。

## 14. 常见故障处理

### 14.1 远程主网页没更新

检查顺序：

1. 是否 commit 了。
2. 是否 push 成功。
3. `deploy_and_verify.js` 是否通过。
4. 远程是否缓存旧版本，使用 `?v=<commit>` 打开。
5. `content_version` 是否同步更新。

### 14.2 页面空白

立即检查：

```bash
node validate_index_html.js
```

如果语法错误，修复后再提交。若已推远程，立即回滚最近提交或修复后快速推送。

### 14.3 财经市场新闻为 0

检查：

1. `news_pool.json` 中 stock/finance 是否有数据。
2. `main_news_refresh.py` 的 `STOCK_REQUIRED_PATTERNS` 是否过窄。
3. `STOCK_OFFTOPIC_PATTERNS` 是否误杀。
4. `apply_news.js` 是否写入 `mockStockNews` 和 `mockStockNewsExtra`。

原则：财经市场宁可保留少量权威财经新闻，也不能显示 0。

### 14.4 首页旧闻太多

检查：

1. `parse_time()` 是否能解析新闻时间。
2. `age_hours()` 是否返回 999。
3. `sort_section()` 是否把权威 pin 放在时效前面过重。
4. `index.html` 前端 `parseNewsTimeValue()` 是否能解析 `7月5日 08:47`、`07-05 08:47`、`2026-07-05 08:47`。

### 14.5 灾害新闻霸屏

立即检查：

1. `main_news_refresh.py` 是否有 `disaster_pin`。
2. `index.html` 是否有 `return 5.2` 或 `freshnessScore * 2.5`。
3. `balanceBreakingDisasterItems()` 是否限制前 20 条最多 2 条。
4. 同一灾害事件是否被聚类去重。

### 14.6 APK 弹升级

检查：

1. 是否误改了 `version` 或 `app_version`。
2. 是否只应该改 `content_version`。
3. `version.json` 中 APK 版本应保持稳定。

## 16. 最后提醒

这套站点最容易出问题的不是“抓不到新闻”，而是排序权重走偏：

- 权威过强，会旧闻霸屏。
- 灾害过强，会首页变灾害专题。
- 过滤过强，会财经/娱乐/足球变 0。
- 去重过强，会误删不同新闻。
- 不做远程验收，会本地已更新但线上没更新。

马维斯接手后，每次最终回复用户时，至少报告四件事：

1. 是否已推送远程。
2. commit 号。
3. 内容版本号。
4. 排序抽样结果，例如：前 20 条中 24 小时内新闻数量、灾害数量、财经市场数量。
